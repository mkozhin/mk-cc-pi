# Airflow DAG Reviewer

Скилл для статического анализа Apache Airflow DAG-файлов (2.9+, ветка 2.x).
Проверяет корректность, best practices, безопасность и выдаёт структурированный отчёт.

## Workflow

### Шаг 1 — Получить код DAG

Код может прийти двумя способами:
- **Путь к файлу**: запусти `scripts/analyze_dag.py` с аргументом `--path`
- **Код в чате**: сохрани во временный файл, затем запусти скрипт

```bash
# Вариант А — файл
python3 /path/to/skill/scripts/analyze_dag.py --path /path/to/dag.py

# Вариант Б — код вставлен в чат (сначала сохрани)
cat > /tmp/_dag_review_input.py << 'DAGEOF'
<вставь код>
DAGEOF
python3 /path/to/skill/scripts/analyze_dag.py --path /tmp/_dag_review_input.py
```

> Путь к самому скиллу найди через `available_skills` — он указан рядом с именем скилла.
> Скрипт лежит в `scripts/analyze_dag.py` относительно папки скилла.

### Шаг 2 — Разобрать JSON-вывод

Скрипт возвращает JSON со структурой:
```json
{
  "score": 72,
  "grade": "Хорошо",
  "issues": [
    {
      "severity": "critical|warning|info|ok",
      "category": "Безопасность",
      "title": "Hardcoded пароль",
      "detail": "Пояснение что не так",
      "fix": "Как исправить"
    }
  ],
  "dag_found": true,
  "syntax_ok": true
}
```

### Шаг 3 — Сформировать отчёт

Используй JSON как основу, но **добавь свой анализ и контекст**:

1. **Заголовок** с источником и итоговой оценкой (score/100 + шкала прогресса)
2. **Таблица** критических / предупреждений / инфо / хорошие практики
3. **Секция ✅ Что сделано правильно** (если есть `ok`-issues)
4. **Секция 🔴 Критические проблемы** — каждая с объяснением *почему* это важно
5. **Секция 🟡 Предупреждения**
6. **Секция 🔵 Рекомендации**
7. **Резюме** — 2-3 предложения о состоянии DAG и приоритетах
8. **Ссылки** на документацию (см. `references/best-practices.md`)

### Шаг 4 — Углублённый анализ (опционально)

Если скрипт нашёл проблемы — дополни их **своим рассуждением**:
- Объясни последствия в контексте этого конкретного DAG
- Предложи конкретный исправленный код (не абстрактный пример)
- Укажи на взаимосвязи между проблемами (например: нет retries + нет идемпотентности = дата-потери)

## Формат score bar

```
████████████░░░░░░░░ 62%
```
- 20 символов: `█` заполненные (score // 5), `░` пустые
- Оценки: ≥85 🏆 Отлично, ≥70 👍 Хорошо, ≥50 ⚠️ Требует улучшений, <50 🚨 Критические проблемы

## Важные правила

- **Не пропускай контекст**: если DAG явно ETL-пайплайн для финансовых данных — акцентируй на idempotency и безопасности сильнее
- **Конкретные фиксы**: показывай исправленный код, не абстрактные советы
- **Приоритизируй**: начинай с критических, не засыпай мелочами если есть серьёзные баги
- **Русский язык**: весь отчёт на русском, технические термины (XCom, TaskFlow, DAG) не переводи
